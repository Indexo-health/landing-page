# Bevel.health 参考总结与修改意见（供审核）

> 目的：吸收 Bevel 的优点，不照抄视觉；用于优化 Indexo Health 的 landing page。

## Bevel 的关键优点（可借鉴）
- **Hero 结构清晰**：一句强主张 + 一段简短说明 + 单一主 CTA；信息密度低但记忆点强。
- **大图主导氛围**：Hero 以全宽大图塑造情绪与质感，文本区独立且留白充足。
- **强视觉节奏**：大图 → 分节内容 → 产品演示 → 生活场景 → 口碑 → CTA 的叙事路径清晰。
- **模块化卡片一致性**：卡片圆角/边框/阴影统一；视觉“家族感”强。
- **信任结构完善**：有 App/产品背书、用户评价、FAQ、合规/法律入口。
- **导航和页脚信息完整**：顶部导航简洁，页脚分列清晰（Company / Product / Resources / Legal + 社媒）。
- **动效克制但有质感**：大图、手机 mock、轻微视差、进入动画增加高级感但不喧宾夺主。

## 可迁移的方向（不照抄）
- **更强 Hero 叙事**：让首页“情绪 + 价值 + 视觉”同时成立，减少冗长解释。
- **更明确的分节节奏**：每节给出“标题/一句话/要点/图”的固定节奏，避免信息混杂。
- **统一卡片语言**：统一圆角/阴影/边框/留白比例，减少模块“风格漂移”。
- **信任与合规结构前置**：把“风险说明/合规文案”放在“Trust”附近，避免页尾的生硬警告。
- **CTA 专注**：每个大节只保留一个主要 CTA，减少多按钮干扰。
- **视觉层次强化**：大标题、轻正文、短要点，减少长段文字。
- **界面证据强化**：用 3–5 张关键界面/报告图，把抽象承诺变成“看得见的结果”。
- **价值四宫格结构**：用 2x2 模块拆解复杂价值（每块只讲一个结果）。
- **AI 服务人格化**：用“对话式预览”展示系统如何解释与提醒，而不是只说“AI”。

## 对 Indexo 页面可执行的修改意见（建议优先级）
### P0（结构与信息）
1. **Hero 重新分层**：
   - 主标题 + 一句价值解释 + 单一 CTA。
   - 右侧/背景主视觉完整呈现（避免遮挡）。
2. **“问题/解决/价值”三段彻底区分**：
   - Problem 只讲痛点；Solution 讲 Pad + Ring；Value 讲“确定的安心/家属通知/晨间简报”。
3. **统一文案密度**：
   - 每个区块不超过 2 行说明；把长文本拆成 2–3 条短 bullet。
4. **四宫格价值拆解**：
   - 睡眠质量 / 呼吸与缺氧 / 心脏与恢复 / 家属安心，形成 2x2 结构。

### P1（视觉与品牌）
1. **统一卡片规范**：
   - 统一圆角（例如 24px/32px）、阴影强度、边框色；所有卡片沿用同一“组件语言”。
2. **配色更有层次**：
   - 以品牌主色 + 中性色层级做“标题/正文/辅助”三档；减少大面积纯色块。
3. **更有节奏的留白**：
   - 标题区与内容区间距加大，区块之间留白更明显。
4. **截图呈现一致**：
   - 统一手机框/透视角/背景色，避免“拼贴感”。

### P2（信任与转化）
1. **引入“背书/引用/评语”模块**：
   - 2–3 条短句式评价或“适合谁”短卡。
2. **页脚完善**：
   - Company / Product / Resources / Legal + 联系方式（已加入结构）。
3. **合规说明专业化**：
   - 改为正式医疗免责声明，不使用“placeholder”。
4. **重复 CTA 但样式一致**：
   - Hero / 中段 / 页尾三处 CTA 保持同一视觉体系。

## 站点与内容规划建议（关于 About / Blog / 文档）
### 是否新增 About / Blog？
**建议新增**，原因：  
- About 用于建立世界观、价值观与专业可信度（信任资产）。  
- Blog 用于持续输出内容（产品迭代、健康科普、研究引用），形成长期 SEO 与可信内容库。  

### 部署方式建议（结合 Vercel 免费托管）
**推荐方案：在现有 Vercel 上直接创建 About / Blog 页面**  
- **优点**：统一品牌与视觉、发布最便捷、无需引入外部平台。  
- **适合**：内容量较少或更新频率低（每月 1–2 篇）。  

**对比 Bevel（Cloudflare + Webflow）的优势**  
- **Webflow**：可视化编辑、非工程人员维护、更新快。  
- **Cloudflare**：CDN/安全/缓存更强，但对我们当前阶段不是刚需。  
- **劣势**：需要额外平台与费用，维护分散。  
结论：**我们直接在 Vercel 发布更便捷，等内容规模变大再迁移**。  

### 文档与法务页面建议（FAQ / Getting Started / Privacy / Terms / Release Notes）
**推荐方案：使用在线文档平台（独立维护）**  
- **优点**：主站与文档分离，更新更独立，非工程同事可直接维护。  
- **可选平台**：  
  - **Craft.do**（成本较低，上手快，适合快速上线）  
  - **Notion**（编辑体验好，结合自定义域名使用）  
  - **Coda**（结构化更强，适合复杂内容）  
- **不建议 Mintlify 付费版**：成本高，当前阶段投入产出不匹配。  

**落地结构建议**  
- `docs.yourdomain.com` 指向文档站  
- 文档站包含：FAQ / Getting Started / Privacy Policy / Terms / Release Notes  

**推荐落地路径（简化）**  
1) **主站**（Vercel）：about-us / blog / landing / 联系方式  
2) **文档站**（Craft 或 Notion）：FAQ / Getting Started / Privacy / Terms / Release Notes  
3) **统一入口**：主站页脚链接到 docs 子域名

## 执行计划（建议）
### Phase 0：内容与信息准备
1) 明确 About 页面核心叙事（使命/原则/团队/合规立场）。
2) 明确 Blog 初期主题与频率（例如每月 1–2 篇）。
3) 收集可公开的信任材料（数据、合作、媒体、用户证言）。

### Phase 1：站点结构与路由
1) 在主站新增 `/about-us` 页面。
2) 在主站新增 `/blog` 列表页（可先是占位与结构）。
3) 预留 `/blog/[slug]` 文章模板（可先用假数据）。
4) 页脚与导航新增 About / Blog / Docs / Contact 入口。

### Phase 2：核心页面改造（Landing）
1) 重构 Hero：更强主张 + 单 CTA + 视觉完整呈现。
2) 调整信息架构：Problem / Solution / Value 分离清晰。
3) 建立 2x2 价值四宫格 + 界面证据墙。
4) 加入 AI 对话预览（2–3 条示例）。

### Phase 3：视觉统一
1) 统一卡片样式（圆角/边框/阴影/留白）。
2) 统一字体层级与图标风格。
3) 统一截图呈现（手机框/角度/背景）。

### Phase 4：信任与转化
1) 引入社媒链接（Instagram / X / Reddit / YouTube / Facebook / Discord）。
2) 增加背书或评价模块（2–3 条）。
3) 完善合规文案与免责声明。

### Phase 5：文档站
1) 选定文档平台（建议 Craft 或 Notion）。
2) 搭建 docs 子域名并发布 FAQ / Getting Started / Privacy / Terms / Release Notes。
3) 主站页脚链接到 docs 子域名。
